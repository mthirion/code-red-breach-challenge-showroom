=  Grizzly credentials

== Principle

The hacker exploited the EventListener, which works based on the "Implicit Trust" of GIT Interceptors and in unauthenticated by default.

The hacker sent a fake json payload to the EventListener, replacing the original GIT repository URL with its own URL.

Even though the Pipeline is configured with Kubernetes Auth Secrets to access GIT repositories in SSH, it's based on the URL, and there is no specific SSH checks if this URL starts with 'http://'


The source code cloned by the pipeline is similar to the one of the original application.
But the pom.xml was modified.  It now contains the Maven Exec Plugin, bind into the compile phase.
As the pipeline ran, the "maven" phase execute "mvn package", which executed the plugin.
What the plugin did was copying the pod token from /var/run/secrets/kubernetes.io/serviceaccount/token into the source code directory of the application.
The maven "package" and "build and sign image" phases dit the rest.
Ad the end, file could have been read from within the application and sent over to a remote server.

The hacker application behaves exactly as the original one, except that it sends sensitive information about the customers to a remote server.

So the source code was slightly modified, the pom.xml was modified, but that's not all that was modified...

=== Pipeline Token
In a standard Tekton pipeline, all the tasks run with the same, 'pipeline' serviceaccount.
This service account is highly permissive, and has the cluster editor permission within the boundary of the namespace.
It therefore allowed the hacker to delete the image of the application used before "leaving".

----
kind: RoleBinding
metadata:
  name: openshift-pipelines-edit
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: edit
subjects:
- kind: ServiceAccount
  name: pipeline
  namespace: hackerapp
----
